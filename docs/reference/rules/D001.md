## D001 - Ảnh hưởng do thay đổi Kiểu Dữ liệu / Độ dài trường (Data Type / Field Length changes)

* **ID:** `D001`
* **Nội dung lỗi (エラー内容):** Thay đổi độ dài/kiểu dữ liệu của các trường chuẩn SAP (ví dụ: `MATNR` 18 -> 40 ký tự) gây ảnh hưởng đến báo cáo, file đầu ra cố định, màn hình chọn, gọi BAPI, subroutine (FORM), hoặc bảng tùy chỉnh (Custom Table).
* **Cách khắc phục (対応方法):**

> [!IMPORTANT]
> **Quy tắc ưu tiên mới:** Với các task D001 mà cách chỉnh sửa chỉ là thêm biến trung gian **dùng để tính toán** nhằm chuyển đổi/giới hạn độ dài hoặc kiểu dữ liệu (ví dụ cộng/trừ `13.2` với `23.2`), **không cần thực hiện sửa này**, vì sửa như vậy không có tác dụng xử lý nguyên nhân gốc.
>
> Khi trả lời cho các case này, output phải nêu rõ: `D001 - No code change. Intermediate-variable fix is not required. Confirm with Teamlead if field length/Data Element impact exists.`
>
> **Không sử dụng `TRY...CATCH`** để xử lý các cảnh báo D001 về tính toán chênh lệch độ dài/kiểu dữ liệu nếu mục đích chỉ là né lỗi ATC hoặc thay thế biến trung gian dùng để tính toán.

### 3.1. Đối với lệnh xuất dữ liệu (`WRITE`)

* Tránh việc độ dài trường mới lớn hơn làm đè hoặc mất nội dung trường bên cạnh.
* **Luôn ưu tiên giữ length/scale theo ECC** cho output, ALV, Add-on/custom table, move assignment và field hiển thị nếu thiết kế không yêu cầu mở rộng. Không tự đổi từ ECC length sang S/4 length chỉ để khớp source.
* Với lỗi D001 liên quan `MATNR` 40 ký tự và field custom/ECC 18 ký tự, **không tự đề xuất** `CONVERSION_EXIT_MATN1_INPUT` hoặc `CONVERSION_EXIT_MATN1_OUTPUT` nếu rule không yêu cầu. Giữ length ECC; nếu dữ liệu thực tế nằm trong phạm vi ECC thì giữ logic và dùng pseudo-comment đúng ATC Note/message sau khi review, nếu không thì confirm Teamlead.
* Với `WRITE` hoặc output hiển thị dạng list/fixed format, **được phép fix độ dài hiển thị** để giữ format ECC cũ, ví dụ giữ độ rộng tương đương `13.2` hoặc độ rộng cột cũ theo thiết kế.
* **Cũ:**

  ```abap
  WRITE: 106 LTH_OUTM-WRBTR CURRENCY W_WAERS, 125 LTH_OUTM-SGTXT.
  WRITE: /3 ST_ZFIA0015-MATNR.
  ```

* **Mới (Chỉ định rõ độ rộng xuất dữ liệu):**

  ```abap
  WRITE: 106(18) LTH_OUTM-WRBTR CURRENCY W_WAERS, 125 LTH_OUTM-SGTXT.
  WRITE: /3(18) ST_ZFIA0015-MATNR.
  ```

> [!IMPORTANT]
> **Quy định dự án:** Được phép sử dụng pseudo-comment khi đã kiểm tra và xử lý code adaptation cần thiết nhưng ATC vẫn yêu cầu xác nhận usage theo SAP Note hoặc message cụ thể (ví dụ `"#EC CI_USAGE_OK[2438131]`, `"#EC CI_FLDEXT_OK[2610650]`). Không dùng pseudo-comment để che lỗi khi chưa phân tích nguyên nhân hoặc khi còn field/interface cần sửa. Ngoại lệ "không sửa bằng biến trung gian" chỉ áp dụng cho case tính toán số học theo quy tắc ưu tiên D001 ở trên.
> Với pseudo-comment DB operation cho `FROM BSEG "#EC CI_DB_OPERATION_OK[2431747]`, phải bọc dòng cũ/dòng mới bằng `S/4HANA Migration UPD START/END` vì đây là thay đổi code để xử lý ATC/S/4HANA compatibility. Không thêm pseudo-comment này vào `FROM BSIS` hoặc `FROM BSAS`.

#### Chi tiết hướng dẫn xử lý lỗi ATC

Khi thực hiện kiểm tra kiểm thử ATC (ABAP Test Cockpit) đối với các trường đã mở rộng độ dài (ví dụ: `MATNR` từ 18 ký tự lên 40 ký tự):

1. **Cảnh báo ATC (`WRITE issue` hoặc `WRITE-Length issue`)**: Xảy ra khi sử dụng lệnh `WRITE` xuất dữ liệu của trường mà không giới hạn độ rộng hiển thị hoặc sau khi chỉ định độ rộng hiển thị (như `(18)`), ATC sẽ đưa ra cảnh báo về nguy cơ mất mát dữ liệu do bị cắt bớt hiển thị.
2. **Cách khắc phục chuẩn (Quy định dự án)**:
   * Sử dụng cú pháp chỉ định độ rộng hiển thị trực tiếp trên lệnh `WRITE` (List Layout Formatting): `WRITE: /3(18) ST_ZFIA0015-MATNR.`
   * **Ngoại lệ (Phương châm cơ bản):** Nếu từ ECC code cũ đã xử lý cố định độ rộng hiển thị theo chuẩn 13.2 cũ (ví dụ có chỉ định sẵn độ rộng `(18)` hoặc tương đương để căn lề hiển thị) thì **giữ nguyên, không cần chỉnh sửa** vì kích thước hiển thị vẫn giống cũ.
   * Nếu đã giữ/fix đúng độ rộng hiển thị theo thiết kế nhưng ATC vẫn yêu cầu xác nhận, có thể thêm pseudo-comment đúng SAP Note/message, ví dụ `"#EC CI_FLDEXT_OK[2610650]`.

> [!WARNING]
> **Tránh sử dụng cú pháp substring `/3 ST_ZFIA0015-MATNR(18)`:**
>
> * **`/3(18) ST_ZFIA0015-MATNR` (Khuyên dùng/Ưu tiên của dự án)**: Đây là cú pháp thiết lập độ rộng hiển thị của cột trên màn hình báo cáo (List Layout Formatting). Cú pháp này **cho phép thực thi Conversion Exit** trước khi hiển thị.
> * **`/3 ST_ZFIA0015-MATNR(18)` (Tránh sử dụng)**: Đây là cú pháp lấy substring vật lý (offset/length access) từ biến `MATNR`. Cú pháp này **bỏ qua Conversion Exit**, dẫn đến việc hiển thị thô chuỗi lưu trữ bao gồm cả các số 0 ở đầu. Ngoài ra, cú pháp này sẽ kích hoạt lỗi cảnh báo ATC mức độ nghiêm trọng liên quan đến truy cập offset/length trên trường có độ dài mở rộng.

### 3.2. Đối với màn hình chọn (`SELECTION-SCREEN`)

* Sử dụng `VISIBLE LENGTH` để giới hạn độ hiển thị màn hình tránh việc đè chữ.
* **Cũ:** `PARAMETERS: P_MATNR TYPE MATNR.`
* **Mới:** `PARAMETERS: P_MATNR TYPE MATNR VISIBLE LENGTH 18.`

### 3.3. Đối với Subroutine (FORM) - Tránh lỗi kiểu dữ liệu không tương thích

* Ngoại lệ "không sửa bằng biến trung gian" chỉ áp dụng khi lỗi thuộc nhóm **tính toán số học**.
* Với lỗi truyền tham số không phải tính toán, không áp dụng cảnh báo no-code-change theo rule tính toán; cần đối chiếu rule chi tiết hoặc confirm Teamlead nếu liên quan đến thay đổi độ dài field/Data Element.

### 3.4. Đối với các kiểu dữ liệu cũ bị thay thế

* Chuyển đổi các kiểu dữ liệu cũ sang kiểu mới trên S4HANA (Ví dụ: `VBTYP_N` chuyển sang `VBTYPL_N`).
* **Cũ:** `VBTYP_N TYPE VBTYP_N,`
* **Mới:** `VBTYP_N TYPE VBTYPL_N,` *(Nếu định nghĩa dạng `TYPE VBFA-VBTYP_N` thì không cần đổi vì DB đã tự cập nhật).*
* Với data element generic `CHAR02` bị ATC báo `Functionality unavailable`, đổi sang data element chuẩn `CHAR_02` thay vì built-in `TYPE C LENGTH 2`.
  * **Cũ:** `LV_MONTH TYPE CHAR02,`
  * **Mới:** `LV_MONTH TYPE CHAR_02,`

### 3.5. Đối với tham số BAPI cập nhật trường dài

* Chuyển các tham số ngắn sang tham số dài (ví dụ: `MATERIAL` thành `MATERIAL_LONG` trong BAPI kiểm tra tồn kho).
* **Cũ:**

  ```abap
  CALL FUNCTION 'BAPI_MATERIAL_AVAILABILITY'
    EXPORTING
      MATERIAL = WK_MATNR
  ```

* **Mới:**

  ```abap
  CALL FUNCTION 'BAPI_MATERIAL_AVAILABILITY'
    EXPORTING
      MATERIAL_LONG = WK_MATNR
  ```

### 3.6. Truy vấn DB (`SELECT`) với trường đổi độ dài (ví dụ `MATNR` 18 -> 40)

* Ngoại lệ "không sửa bằng biến trung gian" không áp dụng cho truy vấn DB không có tính toán số học.
* Nếu lỗi liên quan đến thay đổi độ dài field/Data Element trong điều kiện `WHERE`, cần đối chiếu rule chi tiết hoặc confirm Teamlead trước khi quyết định sửa.

### 3.7. Ảnh hưởng đến Bảng Tùy chỉnh (Custom/Add-on Tables) và ALV

* **Phương châm cơ bản (Quy định mới):** Nếu Addon Table hoặc cấu trúc ALV hiện đang dùng kiểu dữ liệu có độ dài 13.2 thì **giữ nguyên, không cần thay đổi** sang kiểu mới 23.2.
* **Lý do:** Giá trị tiền thực tế từ ECC đến nay không tăng đột biến, nên độ dài 13.2 vẫn đáp ứng đủ thực tế và không xảy ra lỗi tràn số (overflow) ở runtime.
* **Không thêm `TRY...CATCH`** cho ALV/Add-on table chỉ để xử lý chênh lệch `13.2` và `23.2`.

### 3.8. Đối với Domain hoặc Data Element chuẩn bị xóa trên S/4HANA

* Khi một số Domain hoặc Data Element chuẩn cũ của SAP ECC bị xóa trên S/4HANA, các đối tượng custom đang tham chiếu đến chúng sẽ bị lỗi mất tham chiếu gốc và không thể kích hoạt (Activation Error).
* **Cách khắc phục:**
  1. Tìm kiếm Domain/Data Element tiêu chuẩn tương đương đang hoạt động trên S/4HANA có cùng thuộc tính kỹ thuật (Data Type và Length).
  2. Mở đối tượng custom trong giao dịch `SE11` và thay đổi trường tham chiếu sang đối tượng mới.
  *Ví dụ: Thay thế Domain cũ `CHAR05` bằng Domain chuẩn tương đương `BP_TEXT5` (Data Type: `CHAR`, Length: `5`).*

### 3.9. Xử lý Material Code khi truyền tham số USING, ghép chuỗi (Concatenate) hoặc hiển thị (WRITE)

Khi cảnh báo D001 liên quan đến `MATNR` nhưng không phải tính toán số học, không áp dụng ngoại lệ no-code-change của nhóm tính toán. Cần xử lý theo rule tương ứng của thao tác cụ thể:

* `WRITE` hoặc output hiển thị: fix độ dài hiển thị để giữ format ECC cũ nếu cần.
* Truyền tham số, ghép chuỗi hoặc substring: đối chiếu rule chi tiết hoặc confirm Teamlead nếu liên quan đến thay đổi độ dài field/Data Element.

### 3.10. WRITE có `UNDER TEXT-Hxx` và field amount bị mở rộng

Khi `WRITE` dùng `UNDER TEXT-Hxx` và bị D001 do field amount bị mở rộng (ví dụ `DMBTR`, `WRBTR`), fix bằng cách thêm width specifier `(N)` **trước tên field** để giữ độ rộng hiển thị ECC, kèm pseudo-comment `"#EC CI_FLDEXT_OK[2610650]`, bọc trong UPD block. Giá trị `N` (thường là `15` hoặc `16`) do người dùng hoặc Teamlead chỉ định.

> [!IMPORTANT]
> **Quy tắc gộp UPD block:** Nếu nhiều dòng `WRITE ... UNDER TEXT-Hxx` liên tiếp cùng cần fix, **gộp tất cả vào 1 cặp `UPD START/END` duy nhất** — toàn bộ dòng cũ comment out ở trên, toàn bộ dòng mới ở dưới. Không tạo UPD block riêng cho từng dòng.

**Pattern 1 dòng đơn:**

```abap
* S/4HANA Migration UPD START
*               field_name   UNDER TEXT-Hxx CURRENCY waers_field,
                (N) field_name   UNDER TEXT-Hxx CURRENCY waers_field,  "#EC CI_FLDEXT_OK[2610650]
* S/4HANA Migration UPD END
```

**Pattern nhiều dòng liên tiếp (gộp 1 block):**

```abap
* S/4HANA Migration UPD START
*               field_A   UNDER TEXT-H07 CURRENCY waers_field,
*               field_B   UNDER TEXT-H08 CURRENCY waers_field,
*               field_C   UNDER TEXT-H09 CURRENCY waers_field.
                (N) field_A   UNDER TEXT-H07 CURRENCY waers_field,  "#EC CI_FLDEXT_OK[2610650]
                (N) field_B   UNDER TEXT-H08 CURRENCY waers_field,  "#EC CI_FLDEXT_OK[2610650]
                (N) field_C   UNDER TEXT-H09 CURRENCY waers_field.  "#EC CI_FLDEXT_OK[2610650]
* S/4HANA Migration UPD END
```

### 3.11. FOR ALL ENTRIES — lỗi compile do type/length mismatch (Exception bắt buộc dùng biến trung gian)

Khi field custom ECC (ví dụ `CHAR 18`) được dùng trong `FOR ALL ENTRIES WHERE` so sánh với field S/4HANA đã mở rộng (ví dụ `MATNR 40`), ABAP compiler báo lỗi `must have the same type and the same length` — đây là **compile error**, không phải chỉ ATC warning — và Dictionary change không được phép (ECC field giữ nguyên). Trong trường hợp này, bảng trung gian với type đúng là **fix bắt buộc và duy nhất**. Đây là ngoại lệ của rule "không cần biến trung gian" vì ABAP FAE bắt buộc cùng type/length ở compile level.

**Cú pháp cũ (旧):**

```abap
SELECT ...
  FROM ...
  FOR ALL ENTRIES IN IT_SOURCE
  WHERE MATNR = IT_SOURCE-MATNR_FR
    AND BWKEY = IT_SOURCE-WERKS.
```

**Cú pháp mới (新):**

```abap
* S/4HANA Migration ADD START
  DATA:
    BEGIN OF LS_KEY,
      MATNR TYPE MATNR,
      BWKEY TYPE BWKEY,
    END OF LS_KEY,
    LT_KEY LIKE TABLE OF LS_KEY.
  LT_KEY = VALUE #( FOR ls IN IT_SOURCE
                    ( MATNR = CONV #( ls-MATNR_FR )
                      BWKEY = ls-WERKS ) ).
  CHECK LT_KEY IS NOT INITIAL.
* S/4HANA Migration ADD END
  SELECT ...
    FROM ...
* S/4HANA Migration UPD START
*    FOR ALL ENTRIES IN IT_SOURCE
*    WHERE MATNR = IT_SOURCE-MATNR_FR
*      AND BWKEY = IT_SOURCE-WERKS.
     FOR ALL ENTRIES IN LT_KEY
     WHERE MATNR = LT_KEY-MATNR
       AND BWKEY = LT_KEY-BWKEY.
* S/4HANA Migration UPD END
```

> `CHECK LT_KEY IS NOT INITIAL.` phải đặt **ngay trước** `SELECT ... FOR ALL ENTRIES IN LT_KEY` để tránh Full Table Scan khi bảng rỗng.
